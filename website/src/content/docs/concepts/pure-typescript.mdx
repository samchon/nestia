# Pure TypeScript Types

> "TypeScript types vanish at runtime. So how can Nestia validate anything from a type alone?"

If you've asked that, this page is for you.

---

## The phrase, decoded

**"Pure TypeScript type"** means *exactly* what it sounds like:

```ts
interface IArticle {
  id: string & tags.Format<"uuid">;
  title: string & tags.MinLength<3>;
  body: string;
}
```

No `class`. No decorator on the field. No separate schema file. The `tags.*` intersections are still **types** — they have no value, no constructor, no metadata. They are markers the TypeScript compiler reasons about and that get erased on emit, just like every other type.

In other validation libraries you write the type **and** a runtime schema:

| Library            | What you write extra                              |
| ------------------ | ------------------------------------------------- |
| `class-validator`  | Decorator calls (`@IsString`, `@MinLength(3)`) on a class. |
| `zod`              | A separate `z.object({...})` and infer the type from it. |
| `ajv` / TypeBox    | A JSON Schema or TypeBox literal that mirrors the type. |
| **Nestia / typia** | Nothing.                                          |

The "extra" disappears with Nestia because the **transformer** does the work at build time.

---

## So how does it run?

When `ttsc` compiles your code, the Nestia transformer looks at every `@TypedBody()`, `@TypedRoute.*()`, `typia.assert()`, etc. and **inlines the validator** for the exact type at that callsite into the emitted JavaScript.

Source:

```ts
@TypedBody() input: IArticle.ICreate
```

Emitted (excerpt — the real thing is longer):

```js
// validation generated AT COMPILE TIME for IArticle.ICreate
const __is = (input) =>
  "string" === typeof input.title
  && 3 <= input.title.length
  && input.title.length <= 50
  && "string" === typeof input.body;
```

The types are gone — but a specialized check that *encodes* those types lives in the bundle. There is no schema lookup, no reflection, no `Object.keys` loop, no recursive walker. It is hand-written code, written for you, against the exact shape of your interface.

That is the trick.

---

## What types are supported?

Almost everything TypeScript can express. A short, honest list:

| Supported                                              |
| ------------------------------------------------------ |
| Primitives, literal types, enums                       |
| Objects (interface, type alias)                        |
| Arrays, tuples (including rest tuples)                 |
| Unions, intersections                                  |
| Discriminated unions                                   |
| Recursive types (`interface ITree { children: ITree[] }`) |
| `null`, `undefined`, optional properties               |
| `Record`, mapped types, template literal types         |
| `string & tags.Format<"...">` and similar tags         |
| Cross-file imports, generic instantiations             |

| Not supported (and what to do instead)                 |
| ------------------------------------------------------ |
| `any` / `unknown` — defeats the point; use a real type. |
| Function / class instance shapes — Nestia validates **data**, not behavior. Use a DTO interface. |

---

## When `class` is fine

You can still use a class as a DTO if your code reads better that way. Nestia treats `class IFoo { … }` the same as `interface IFoo { … }` — it derives the schema from the property types. You just gain nothing extra from `class` over `interface`.

What *won't* work is a class whose properties depend on **constructor metadata** — there is no constructor running at validation time. Keep DTOs as data, put behavior in services.

---

## So the type is the contract

This is the core idea. The TypeScript type is the **only** declaration of the request and response shape. Nestia derives validation, the OpenAPI schema, the SDK function signature, and the e2e fixture from that single source. Change the type — all four follow.

Most "duplication" bugs in API code come from drift between the type, the validator, and the schema. Nestia removes the possibility.
