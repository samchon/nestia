# Why So Fast

Two headline numbers in the Nestia docs:

- Validation is up to **20,000× faster** than `class-validator`.
- JSON serialization is up to **200× faster** than `class-transformer`.

This page explains where those numbers come from and why they matter on a real server.

---

## Where the speedup comes from

In a sentence: Nestia turns generic schema walkers into **specialized, inlined predicates**.

A generic validator runs roughly:

```text
for each field in schema:
    look up rule
    apply rule to value
    if rule depends on nested type:
        recurse
```

Every step is indirection — a property access, a function call, sometimes an `Object.keys` loop, sometimes a `Reflect.getMetadata`. JavaScript engines cannot inline most of this.

Nestia's generated validator runs:

```js
"string" === typeof input.title
  && 3 <= input.title.length
  && input.title.length <= 50
  && "string" === typeof input.body
  && /* … */;
```

It is a flat boolean expression. Modern V8 / SpiderMonkey JIT this aggressively. Cold cost ≈ a few hundred nanoseconds per object.

The same story applies to JSON serialization. `JSON.stringify` is generic; Nestia generates a template-literal encoder that writes out exactly your object's shape.

## The benchmark setup

The benchmarks live in [`typia/benchmark`](https://github.com/samchon/typia/tree/master/benchmark) and [`nestia/benchmark`](https://github.com/samchon/nestia/tree/master/benchmark). They are reproducible — run `pnpm test` after cloning and you get fresh numbers on your hardware.

Each library is fed the same set of payload shapes (simple, hierarchical, recursive, union, tuple, dynamic-keyed). For each shape we run *N* validate / stringify cycles and divide by elapsed time. The graphs you see on this site are from `AMD R9-7940HS`.

A few honest caveats:

- **Synthetic.** Like every microbenchmark, real servers have other costs (DB, network, framework overhead) that compress relative differences.
- **20,000× is the peak**, not the average. On simple objects it is more like **30–100×**. The big numbers come from cases where `class-validator` does extra work (e.g. recursive unions).
- **`fast-json-stringify`** (used by `fastify`) is also fast on stringify — but it needs a hand-written JSON Schema and does not validate input.

The point is not "Nestia wins every microbenchmark." It is that **you get serious validation and serialization speedups for free**, while writing less code.

## Does it show up on a real server?

Yes. Here is the rough math.

Node's HTTP work is non-blocking (the event loop hands I/O to libuv). But **JSON.stringify of the response is synchronous** — it runs on the main thread and blocks the next request. For a server that returns even moderately-sized JSON, response stringification can occupy 20–40% of main-thread time under load.

Replacing it with Nestia's specialized encoder shifts that budget back to your handler. The [`nestia/benchmark`](https://github.com/samchon/nestia/tree/master/benchmark/results/AMD%20Ryzen%209%207940HS%20w%20Radeon%20780M%20Graphics) results page shows a 2–10× RPS gain on representative endpoints — exactly the kind of jump you would see by switching to fastify, but without giving up the NestJS ecosystem.

Validation savings are smaller in absolute terms (request bodies are usually smaller than responses) but they apply on every POST / PUT / PATCH that has a `@TypedBody()`.

---

## When the speedup doesn't matter

Be honest:

- Endpoints whose latency is dominated by a DB query won't get faster.
- Endpoints behind a slow third-party API won't get faster.
- A nightly-cron service that handles 100 requests a day won't notice.

You should pick Nestia for the **type-safety story** first (one source of truth, generated SDK, generated Swagger). The speed is a nice property that happens to come with the architecture, not the reason to adopt it.
