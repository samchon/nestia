# Migrating from `class-validator`

The mechanical part of moving off `class-validator`: a translation table between the most common decorators and `typia.tags`.

You can migrate file by file â€” Nestia ignores `class-validator` decorators that are still there, so the two systems coexist during the transition.

---

## Pattern: class â†’ interface

Before:

```ts
import { IsString, MinLength, MaxLength, IsEmail, IsUUID, IsOptional } from "class-validator";

export class CreateUser {
  @IsEmail()
  email!: string;

  @IsString() @MinLength(8) @MaxLength(64)
  password!: string;

  @IsOptional() @IsString()
  nickname?: string;

  @IsUUID()
  invited_by!: string;
}
```

After:

```ts
import { tags } from "typia";

export interface ICreateUser {
  email: string & tags.Format<"email">;
  password: string & tags.MinLength<8> & tags.MaxLength<64>;
  nickname?: string;
  invited_by: string & tags.Format<"uuid">;
}
```

The class becomes an interface (optional â€” a class still works). Decorators disappear; their constraints live inside the type.

## Decorator mapping

### Strings

| `class-validator`             | Nestia                                          |
| ----------------------------- | ----------------------------------------------- |
| `@IsString()`                 | `string`                                        |
| `@MinLength(3)`               | `& tags.MinLength<3>`                           |
| `@MaxLength(50)`              | `& tags.MaxLength<50>`                          |
| `@Length(3, 50)`              | `& tags.MinLength<3> & tags.MaxLength<50>`      |
| `@Matches(/^[a-z]+$/)`        | `& tags.Pattern<"^[a-z]+$">`                    |
| `@IsEmail()`                  | `& tags.Format<"email">`                        |
| `@IsUrl()`                    | `& tags.Format<"url">`                          |
| `@IsUUID()`                   | `& tags.Format<"uuid">`                         |
| `@IsDateString()` (ISO)       | `& tags.Format<"date-time">`                    |
| `@IsIP()`                     | `& tags.Format<"ipv4">` or `<"ipv6">`           |

### Numbers

| `class-validator`             | Nestia                                          |
| ----------------------------- | ----------------------------------------------- |
| `@IsNumber()`                 | `number`                                        |
| `@IsInt()`                    | `number & tags.Type<"int32">`                   |
| `@IsPositive()`               | `& tags.ExclusiveMinimum<0>`                    |
| `@Min(0)`                     | `& tags.Minimum<0>`                             |
| `@Max(100)`                   | `& tags.Maximum<100>`                           |
| `@IsBoolean()`                | `boolean`                                       |

### Arrays

| `class-validator`             | Nestia                                          |
| ----------------------------- | ----------------------------------------------- |
| `@IsArray()`                  | `T[]` (in the type)                             |
| `@ArrayNotEmpty()`            | `& tags.MinItems<1>`                            |
| `@ArrayMinSize(3)`            | `& tags.MinItems<3>`                            |
| `@ArrayMaxSize(10)`           | `& tags.MaxItems<10>`                           |
| `@ValidateNested({ each: true })` | TypeScript handles it â€” `Foo[]` already validates each element. |

### Nullability and presence

| `class-validator`             | Nestia                                          |
| ----------------------------- | ----------------------------------------------- |
| `@IsOptional()`               | `?` on the property                             |
| Nullable                      | `\| null` in the type                           |
| `@IsDefined()`                | Required property (no `?`)                      |

### Enums and unions

| `class-validator`             | Nestia                                          |
| ----------------------------- | ----------------------------------------------- |
| `@IsEnum(MyEnum)`             | `MyEnum` (TS enum type works directly)          |
| `@IsIn(["a","b"])`            | `"a" \| "b"`                                    |

### Nested objects

| `class-validator`             | Nestia                                          |
| ----------------------------- | ----------------------------------------------- |
| `@Type(() => Child)` + `@ValidateNested()` | `child: IChild` in the type           |

You don't need `@Type` because Nestia reads the static type, not the runtime constructor.

## Custom validators

If you have a `@IsCustomXyz()` decorator, port it to a `tags.Pattern<...>` (for string regex) or to a JSDoc `@type` tag with a custom typia tag.

For more complex business rules â€” "this email must not already exist" â€” the right place is your **handler** or a **NestJS guard**, not the validator. `class-validator`'s async validators are a foot-gun (they run twice for every request); Nestia deliberately doesn't ship that pattern.

---

## What to do about `@Transform`

`class-transformer`'s `@Transform` ran before validation. With Nestia, do the transformation in your handler â€” the validation runs first and you have a typed input from there:

```ts
@TypedRoute.Post()
async create(@TypedBody() input: ICreateUser): Promise<IUser> {
  // input is typed and validated. Transform here, not in the DTO.
  return this.service.create({
    ...input,
    email: input.email.toLowerCase(),
  });
}
```

## When you're done

Remove `class-validator`, `class-transformer`, and `@Type` imports from your codebase. Drop the `ValidationPipe` from `main.ts` â€” Nestia handles the same job inside each `@TypedBody()` / `@TypedQuery()` / `@TypedParam()`.
