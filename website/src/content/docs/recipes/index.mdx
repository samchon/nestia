# FAQ

The questions that come up most often, with answers short enough to scan.

---

### Does Nestia work with NestJS guards, interceptors, pipes, and exception filters?

Yes — all of them. Nestia decorators are normal NestJS decorators that add validation; the rest of the NestJS request pipeline (`@UseGuards`, `@UseInterceptors`, `@UsePipes`, global exception filters) runs around them exactly as you would expect. The order is the standard NestJS order: guards → interceptors → pipes → handler. Nestia validation happens at the pipe stage.

```ts
import { JwtAuthGuard } from "./auth/JwtAuthGuard";

@UseGuards(JwtAuthGuard)
@TypedRoute.Post()
public async create(@TypedBody() input: ICreate): Promise<IArticle> { ... }
```

`@nestjs/passport`, `@nestjs/jwt`, custom guards, and ACL libraries all work unchanged.

---

### How do I get the authenticated user inside a `@TypedRoute` handler?

Same as vanilla NestJS — use a custom `@User()` parameter decorator or `@Request()`. Nestia decorators only cover **typed input** (body, query, param, headers); identity comes from your guard or middleware:

```ts
import { createParamDecorator, ExecutionContext } from "@nestjs/common";

export const User = createParamDecorator(
  (_data, ctx: ExecutionContext) => ctx.switchToHttp().getRequest().user,
);

@UseGuards(JwtAuthGuard)
@TypedRoute.Post()
public async create(
  @User() user: IAuthUser,
  @TypedBody() input: ICreate,
): Promise<IArticle> { ... }
```

The generated SDK skips parameters whose decorators it does not recognize, so `@User` does not leak into the client signature.

### Can I set a global prefix or enable versioning?

Yes. Set them on the app inside `nestia.config.ts`'s `input` function so the generator sees them:

```ts filename="nestia.config.ts"
const config: INestiaConfig = {
  input: async () => {
    const app = await NestFactory.create(AppModule);
    app.setGlobalPrefix("api");
    app.enableVersioning({ type: VersioningType.URI, prefix: "v" });
    return app;
  },
  output: "src/api",
};
```

The generated SDK paths reflect the prefix and version automatically.

### Can I use Prisma or TypeORM models as DTOs directly?

You can, but consider not to. Returning a Prisma `Article` directly leaks the ORM relations into your SDK and forces the frontend to depend on `@prisma/client`. Two clean options:

- **Define DTO interfaces alongside the model**, then map at the controller boundary.
- **Turn on `clone: true`** in `nestia.config.ts` — Nestia copies the types into the SDK's own `structures/` directory, breaking the dependency.

---

### Do I need to replace every `@Get` with `@TypedRoute.Get`?

No. Vanilla NestJS decorators still work. Use `@TypedRoute.*` where you want a typed response and faster serialization; use `@Get` / `@Post` / etc. anywhere else. Nestia's SDK / Swagger / e2e generators only emit code for `@Typed*` routes, so vanilla routes are simply invisible to them.

---

### Can I mix `class-validator` and `@TypedBody()` in the same project?

Yes. They are independent. A controller method can use `@Body()` with `class-validator` next to one that uses `@TypedBody()`. Migrate at whatever pace makes sense.

---

### My existing `@nestjs/swagger` `@ApiProperty()` decorators — do they break?

No, Nestia ignores them. They can stay until you remove them. The migration recipe is in **[Migrating from @nestjs/swagger](/docs/recipes/from-nestjs-swagger)**.

---

### Does `nest build` work?

No. `nest build` invokes the stock TypeScript compiler, which has no plugin system, so the Nestia transformer never runs. Replace it with `ttsc` — see **[Setup → TypeScript-Go](/docs/setup/tsgo#compile)**.

The same goes for `nest start` (use `node dist/main.js` after `ttsc`), `nest start --watch` (use `concurrently "ttsc --watch" "node --watch dist/main.js"`), and `nest build --webpack` (use `npx webpack` with `@ttsc/unplugin/webpack`).

---

### Do I need Go installed?

Yes, on first build. The Nestia transformer ships as a Go binary that builds on demand from the Go sources inside `@nestia/core`. Go 1.26 or newer must be on `PATH`, or pointed at by `TTSC_GO_BINARY`. The binary is cached after the first build.

---

### Does it work with `fastify`?

Yes. Nestia knows nothing about the HTTP adapter — it operates at the decorator and serializer level. NestJS in fastify mode benefits from the faster serializer the same way (and arguably more, because fastify's bottleneck is response serialization).

---

### Does it work with monorepos?

Yes. The common shape is "SDK as a third workspace package" — backend writes the SDK to a sibling workspace, frontend imports it by name. Full walkthrough in **[SDK → PNPM Monorepo](/docs/sdk/monorepo)**.

---

### Does it work with `pnpm` / `yarn berry` / `bun`?

Yes. The `@next` setup wizard supports all four (`npx nestia setup --manager pnpm` etc.). The legacy v11 line does not support yarn berry — see **[Setup → Legacy](/docs/setup/legacy)**.

---

### Can I generate the SDK without starting a NestJS app?

Yes. Set `input` to a glob of controller files instead of a `NestFactory.create` function:

```ts
const config: INestiaConfig = {
  input: { include: ["src/**/*.controller.ts"] },
  output: "src/api",
};
```

The trade-off: this mode doesn't see `app.setGlobalPrefix()` or `app.enableVersioning()`. Prefer the `NestFactory.create` form when your app uses them.

---

### How do I document an endpoint?

JSDoc on the controller method:

```ts
/**
 * Create an article.
 *
 * @param input The payload.
 * @returns The stored article.
 */
@TypedRoute.Post()
public async create(@TypedBody() input: ICreate): Promise<IArticle> { ... }
```

Nestia copies that JSDoc into:

- The Swagger `summary` / `description`.
- The SDK function's JSDoc (so your frontend IDE shows it on hover).
- The e2e test file's header.

Field-level JSDoc on DTO properties does the same for OpenAPI's `description` / `title`.

---

### How do I document an error response?

Use `@TypedException`:

```ts
@TypedException<IBadRequest>(400, "When the payload is malformed")
@TypedRoute.Post()
public async create(...) { ... }
```

See **[Core → TypedException](/docs/core/TypedException)**.

---

### Can I get every validation error, not just the first?

Yes. Switch the validation mode to `"validate"` in `tsconfig.json` — full options on **[Core → TypedBody → Configuration](/docs/core/TypedBody#configuration)**.

---

### Can the SDK throw a typed error instead of an untyped `HttpError`?

Yes. Set `propagate: true` in `nestia.config.ts` and the SDK returns a discriminated union keyed by status code. Combined with `@TypedException`, the frontend can branch on `output.success` / `output.status` with full type narrowing. See **[Tutorial → Call Your API](/docs/tutorial/consume#4-handle-errors)**.

---

### What's the relationship between `@nestia/core` and `typia`?

`@nestia/core` is the NestJS bindings layer. It calls into `typia` for the actual validation and stringification primitives. You install both — `npx nestia setup` does it for you.

---

### Why does my build break after upgrading `typia`?

The `@nestia/core@next` and `typia@next` versions are co-released. Bumping one without the other will fail. Run `npx nestia dependencies` to check, or `npm i typia@next @nestia/core@next @nestia/sdk@next @nestia/fetcher@next` in one shot.
