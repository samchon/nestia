# Production Deployment

A Nestia server in production is just a Node app. The Nestia transformer runs at **build time** — the `dist/` output it produces does not need `ttsc`, `typia`, or Go to run. That keeps deployment boring on purpose.

The actual shipping flow:

1. Build locally or in CI with `ttsc`.
2. Copy `dist/`, `package.json`, and `node_modules` (or bundle them) to the runtime.
3. Run `node dist/main.js`.

This page covers the four shapes that come up most.

---

## The build

A canonical `package.json`:

```jsonc filename="package.json" copy
{
  "scripts": {
    "build": "ttsc && nestia all",
    "start": "node dist/main.js",
    "start:prod": "NODE_ENV=production node dist/main.js"
  }
}
```

`nestia all` runs `sdk + swagger + e2e` after the TypeScript build. You can drop it from the deploy script if you regenerate those artifacts separately.

<Callout type="info">
**CI needs Go.** `ttsc` builds the Nestia transformer from Go source on the first invocation per machine. CI runners that bake the image once and reuse it pay this cost once. Runners that rebuild from scratch every time should either install Go in the workflow (`actions/setup-go@v5`) or cache `$HOME/.cache/ttsc`.
</Callout>

## Docker

The two-stage Dockerfile is the safest pattern: install + build with the full toolchain, then run from a slim image with only production dependencies.

```dockerfile filename="Dockerfile" copy
# ---------- builder ----------
FROM node:22-bookworm AS builder
WORKDIR /app

# Go is required for the Nestia transformer's first build.
RUN apt-get update && apt-get install -y --no-install-recommends \
      golang-1.23 \
    && ln -s /usr/lib/go-1.23/bin/go /usr/local/bin/go \
    && rm -rf /var/lib/apt/lists/*

COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build       # ttsc → dist/
RUN npm prune --omit=dev

# ---------- runtime ----------
FROM node:22-bookworm-slim AS runtime
WORKDIR /app
ENV NODE_ENV=production

COPY --from=builder /app/dist          ./dist
COPY --from=builder /app/node_modules  ./node_modules
COPY --from=builder /app/package.json  ./

EXPOSE 3000
CMD ["node", "dist/main.js"]
```

The runtime image has no Go and no TypeScript compiler — only the compiled JS and pinned `node_modules`.

<Callout type="warning">
**Go's apt name moves.** `golang-1.23` was current at the time of writing. Pin whichever version `apt search golang-1` lists in your base image — the transformer needs Go **1.26+** for the latest features but is compatible with **1.23+** on older binaries.
</Callout>

## AWS Lambda / single-file bundle

Lambda dislikes large `node_modules`. Bundle the server into one JS file with Webpack or esbuild — `@ttsc/unplugin` runs the Nestia transformer inside the bundler.

<Tabs items={["Webpack + ts-loader", "esbuild"]}>
  <Tabs.Tab>
```js filename="webpack.config.mjs" copy
import { fileURLToPath } from "node:url";
import nodeExternals from "webpack-node-externals";
import ttsc from "@ttsc/unplugin/webpack";

export default {
  entry: "./src/main.ts",
  output: {
    path: fileURLToPath(new URL("./dist", import.meta.url)),
    filename: "main.js",
    libraryTarget: "commonjs2",  // for Lambda
  },
  target: "node",
  mode: "production",
  externals: [nodeExternals()],
  module: { rules: [{ test: /\.ts$/, loader: "ts-loader" }] },
  resolve: { extensions: [".ts", ".js"] },
  plugins: [ttsc()],
};
```

For a fully bundled single file (no `node_modules` in the deploy package), see [Setup → Legacy → Single JS file only](/docs/setup/legacy#single-js-file-only) — same recipe applies to `@next`.
  </Tabs.Tab>
  <Tabs.Tab>
```ts filename="esbuild.config.ts" copy
import ttsc from "@ttsc/unplugin/esbuild";
import { build } from "esbuild";

await build({
  entryPoints: ["src/main.ts"],
  outfile: "dist/main.js",
  platform: "node",
  bundle: true,
  // NestJS optional peers — leave external or `IgnorePlugin` equivalent.
  external: [
    "@nestjs/microservices", "@nestjs/websockets",
    "@fastify/static", "@fastify/view",
    "class-transformer", "class-validator",
  ],
  plugins: [ttsc()],
});
```
  </Tabs.Tab>
</Tabs>

Wrap the bundled handler in your platform's Lambda adapter — `serverless-http`, `@vendia/serverless-express`, or your provider's native shim. Nestia is not involved at runtime.

## Vercel / Netlify (NestJS in a serverless function)

Vercel and Netlify expect a function handler, not a long-running server. The pattern is the same as Lambda: bundle, then re-export through an adapter.

```ts filename="api/[[...slug]].ts" copy
import { NestFactory } from "@nestjs/core";
import serverlessExpress from "@vendia/serverless-express";

import { AppModule } from "../src/AppModule";

let cached: any;

export default async (req: any, res: any) => {
  if (!cached) {
    const app = await NestFactory.create(AppModule);
    await app.init();
    cached = serverlessExpress({ app: app.getHttpAdapter().getInstance() });
  }
  return cached(req, res);
};
```

The build still needs `ttsc`. Set `vercel.json`'s `buildCommand` to `npm run build` and ship the `dist/` folder.

## Fly.io / Railway / Render (long-running server)

These platforms just want a Dockerfile or a `Procfile`. The Dockerfile above works as-is for Fly and Railway; for Render, point the start command at `npm run start:prod`.

## Where the SDK gets published

Two patterns:

- **Repo-internal SDK.** `output: "src/api"` in `nestia.config.ts`. Frontend lives in the same repo and imports from `../api`. No publish step needed.
- **Standalone SDK package.** `distribute: "packages/api"`. After `ttsc && nestia sdk`, `cd packages/api && npm publish`. Partners install `@your-org/api` from npm. See **[SDK → Distribution](/docs/sdk/distribute)**.

## Health checks

A typical health endpoint that survives validation:

```ts filename="src/health/HealthController.ts" copy
import { TypedRoute } from "@nestia/core";
import { Controller } from "@nestjs/common";

@Controller("health")
export class HealthController {
  @TypedRoute.Get()
  public async check(): Promise<{ ok: true; uptime: number }> {
    return { ok: true, uptime: process.uptime() };
  }
}
```

Use it as the cloud platform's liveness probe (`/health`).

## Common deploy-time errors

- **`go: command not found`** in CI. Add Go to the runner image, or skip `ttsc` in the deploy stage by building locally and uploading `dist/`.
- **`Cannot find module '@nestjs/microservices'`** in a bundled Lambda. Add it to `externals` / `IgnorePlugin` — it is an optional peer of NestJS, used only if you import it.
- **`Error on nestia.core.<method>(): no transform has been configured`** at runtime. The build did not run through `ttsc`. Almost always a leftover `nest build` somewhere in the deploy pipeline — see [Troubleshooting](/docs/recipes/troubleshooting).
