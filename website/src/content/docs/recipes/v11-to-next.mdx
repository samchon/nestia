# v11 → `@next` Migration

If your project is on **`nestia@^11`** (legacy line, `ts-patch`-based) and you want to move to **`@next`** (Go transformer, `ttsc`-based), this is the cutover.

It is **reversible** — keep your previous lockfile and `tsconfig.json` in version control until the new build is green. Six steps, roughly 30 minutes for a typical project.

<Callout type="warning">
The `@next` line is a preview track. `npm` tag `next` moves with every dev release. If you need a reproducible build, pin to an exact dev version (e.g. `typia@13.0.0-dev.20260514`) instead of `@next`.
</Callout>

---

## 1. Clear stale outputs

A fresh build must not run on top of artifacts from the legacy transformer:

```bash filename="Terminal" copy
rm -rf dist/ lib/
```

---

## 2. Drop the `ts-patch` chain

`ts-patch` patched the JavaScript TypeScript compiler in place. `@next` uses an unpatched compiler and a Go binary; the patch must go.

```bash filename="Terminal" copy
npm uninstall ts-patch
```

Remove the `prepare` script from `package.json`:

```diff filename="package.json"
 {
   "scripts": {
-    "prepare": "ts-patch install",
     ...
   }
 }
```

If you historically ran `npx typia patch`, drop that too — the `@next` runtime does not need it and a patched `typia` will conflict with the Go-emitted code.

---

## 3. Install `@next` packages

This is the exact order the `npx nestia setup` wizard uses on `@next`. The blank line splits the typia / compiler layer from the framework layer.

<Tabs items={["npm", "pnpm", "yarn", "bun"]}>
  <Tabs.Tab>
```bash filename="Terminal" copy
npm i -D ttsc @typescript/native-preview
npm i typia@next

npm i @nestia/core@next @nestia/sdk@next @nestia/fetcher@next
npm i -D nestia@next
```
  </Tabs.Tab>
  <Tabs.Tab>
```bash filename="Terminal" copy
pnpm i -D ttsc @typescript/native-preview
pnpm i typia@next

pnpm i @nestia/core@next @nestia/sdk@next @nestia/fetcher@next
pnpm i -D nestia@next
```
  </Tabs.Tab>
  <Tabs.Tab>
```bash filename="Terminal" copy
yarn add -D ttsc @typescript/native-preview
yarn add typia@next

yarn add @nestia/core@next @nestia/sdk@next @nestia/fetcher@next
yarn add -D nestia@next
```
  </Tabs.Tab>
  <Tabs.Tab>
```bash filename="Terminal" copy
bun add -d ttsc @typescript/native-preview
bun add typia@next

bun add @nestia/core@next @nestia/sdk@next @nestia/fetcher@next
bun add -d nestia@next
```
  </Tabs.Tab>
</Tabs>

Then refresh the lockfile so old resolutions don't survive:

```bash filename="Terminal" copy
rm pnpm-lock.yaml package-lock.json yarn.lock bun.lockb 2>/dev/null
npm install     # or pnpm install / yarn install / bun install
```

Note: `@nestia/sdk` is now a **runtime** dependency, not a dev dependency. The Go binary needs to resolve it whenever the SDK / Swagger / e2e transform attaches metadata.

---

## 4. Simplify `tsconfig.json`

The simplest move is to **delete** `compilerOptions.plugins` entirely — `ttsc` auto-discovers Nestia and typia through their `package.json#ttsc.plugin` markers.

```diff filename="tsconfig.json"
 {
   "compilerOptions": {
     "strict": true,
     "experimentalDecorators": true,
     "emitDecoratorMetadata": true,
-    "plugins": [
-      { "transform": "@nestia/core/lib/transform" },
-      { "transform": "@nestia/sdk/lib/transform" },
-      { "transform": "typia/lib/transform" }
-    ]
   }
 }
```

Keep `experimentalDecorators` and `emitDecoratorMetadata` — NestJS needs them for constructor injection.

<Callout type="info">
**Need transform options** (`stringify`, `llm`, etc.)? Keep an explicit `plugins` array, but the **order changes** on `@next`: `typia` first as a disabled tombstone, then `@nestia/sdk`, then `@nestia/core/native/transform.cjs`. See [Setup → TypeScript-Go → Transform Options](/docs/setup/tsgo#transform-options).
</Callout>

---

## 5. Switch the build commands

Replace every `tsc` / `nest build` with `ttsc`. Replace every `ts-node` / `tsx` with `ttsx`. Use `concurrently` for watch scripts so they behave the same on cmd, PowerShell, and POSIX shells.

```diff filename="package.json"
 {
   "scripts": {
-    "prepare": "ts-patch install",
-    "build": "nest build",
-    "start": "nest start",
-    "start:dev": "nest start --watch",
-    "start:debug": "nest start --debug --watch",
-    "start:prod": "node dist/main"
+    "build": "ttsc",
+    "start": "node dist/main.js",
+    "start:dev": "concurrently \"ttsc --watch\" \"node --watch dist/main.js\"",
+    "start:debug": "concurrently \"ttsc --watch\" \"node --watch --inspect-brk dist/main.js\"",
+    "start:prod": "node dist/main.js"
   }
 }
```

Add `concurrently` as a devDependency:

```bash filename="Terminal" copy
npm i -D concurrently
```

`--inspect-brk` waits for the debugger to attach; switch to `--inspect` for non-blocking attach. If you prefer a single watcher that re-runs the script in-process, use `ttsx --watch src/main.ts`.

---

## 6. Verify

Confirm **Go 1.26+** is on `PATH` (or `TTSC_GO_BINARY` points at a usable binary). Then:

```bash filename="Terminal" copy
npx ttsc --noEmit       # surface any plugin or pin diagnostics first
npx nestia all          # regenerate SDK / Swagger / e2e against the new toolchain
npm run build
npm run start:prod      # confirm runtime works
```

Your existing `nestia.config.ts` is **forward-compatible** — `@nestia/sdk@next` reads the same shape, so you do not need to re-run `npx nestia init` unless you want a fresh starter.

## Common silent failure

The single most common post-migration bug: leaving `nest build` or `nest start` somewhere in `package.json`, `nest-cli.json`, or a CI script.

Both invoke stock `tsc`. The build *appears* to succeed, but the Nestia transform never runs, and the first request to a `@TypedBody` endpoint throws:

```text
Error on nestia.core.TypedBody(): no transform has been configured.
```

Grep for `nest build`, `nest start`, and `builder` (in `nest-cli.json`) before declaring the migration done.

## What changes for your code?

Almost nothing. Decorator names, `INestiaConfig` options, CLI flags, and the generated SDK / Swagger / e2e surface are **all preserved**. The `@next` line is a toolchain migration, not an API migration.

A handful of behavior changes are worth knowing:

- **Default validation mode is `validate`** (not `assert`). Extras pass through silently. Set `validate: "validateEquals"` in transform options to keep the legacy behavior, or `validate: "assertPrune"` to strip extras. See [TypedBody → Configuration](/docs/core/TypedBody#configuration).
- **`@nestia/sdk` is now a runtime dependency** — the Go binary resolves it whenever metadata attaches.
- **Yarn berry is supported** on `@next` (it was not on v11).
- **Bun is supported** on `@next`.

---

## Rolling back

If something breaks and you need to revert quickly:

1. `git checkout` your old `package.json`, `tsconfig.json`, lockfile, and `package-lock.json`.
2. `npm install` to restore the v11 dependency set.
3. `npm run prepare` to re-patch the local TypeScript compiler.
4. `npm run build`.

You should be back on the legacy line in under five minutes.
