# Troubleshooting

The errors people hit, in rough order of frequency.

---

## Build errors

### `Error on nestia.core.<method>(): no transform has been configured`

The transformer never ran. Almost always one of:

1. The build used **`tsc`** instead of `ttsc`. Replace `nest build` / `tsc` in your scripts with `ttsc`. See [Setup тЖТ TypeScript-Go тЖТ Compile](/docs/setup/tsgo#compile).
2. `nest-cli.json` has `compilerOptions.builder: "swc"` or `"webpack"`. That bypasses `ttsc`. Remove the field, or stop using `nest build` and use `ttsc` (or `@ttsc/unplugin/webpack`) directly.
3. `npx nestia setup` was never run тАФ `ttsc` is not installed.

On the legacy `nestia@^11` line, the equivalent failure mode is that `ts-patch install` did not run. Make sure `prepare: "ts-patch install"` is in `package.json` and has executed at least once.

---

### `go: command not found` or a Go diagnostic during build

The Nestia transformer builds its Go binary on demand. Install Go 1.26 or newer from [go.dev/dl](https://go.dev/dl/) and rerun the build. Or set `TTSC_GO_BINARY` to a prebuilt path if you ship a binary internally.

---

### `Cannot find module '@nestia/sdk/lib/transform'` or similar

The plugin entry in `tsconfig.json` points at a path that no longer exists, or you upgraded to `@next` but the lockfile still pins old paths. Two fixes:

- **Easiest:** delete `compilerOptions.plugins` entirely. `ttsc` auto-discovers Nestia and typia through their `package.json#ttsc.plugin` markers тАФ you do not need an explicit array.
- **If you must keep an explicit array** (because you set `stringify` or `llm`), use the path documented in [Setup тЖТ TypeScript-Go тЖТ Transform Options](/docs/setup/tsgo#transform-options).

---

### Mixed `@latest` / `@next` packages

`@nestia/core@next`, `@nestia/sdk@next`, `@nestia/fetcher@next`, `nestia@next`, and `typia@next` are co-released and must move together. A mixed install fails with confusing transform errors. Fix:

```bash
npm i @nestia/core@next @nestia/sdk@next @nestia/fetcher@next typia@next
npm i -D nestia@next
```

Then refresh the lockfile so old resolutions don't survive: `rm package-lock.json && npm install` (or the equivalent for pnpm / yarn / bun).

---

### Bundler doesn't apply the transform

`Vite`, `Next.js`, `esbuild`, `Bun.build`, `Rollup`, `Rolldown`, `Farm`, `Rspack` тАФ none of them use the TypeScript compiler by default, so they skip `ttsc` plugins entirely. Install `@ttsc/unplugin` and add the matching adapter to your bundler config. See [Setup тЖТ TypeScript-Go тЖТ Bundlers](/docs/setup/tsgo#bundlers).

If you're bundling a NestJS server (not a frontend), additionally make sure your TS pipeline emits decorator metadata тАФ only `ts-loader` and (with opt-in flags) `builtin:swc-loader` do.

## Runtime errors

### 400 Bad Request on a payload that "looks right"

Open the response body тАФ Nestia returns `{ statusCode: 400, message, errors: [{ path, expected, value }] }` (default `validate` mode lists every failure). The `path` on each entry points at a failing field. Common culprits:

- **Extra properties.** Default mode (`validate`) lets unknown properties pass through silently. Use `validateEquals` to reject them, or `validatePrune` / `assertPrune` to strip them. See [TypedBody тЖТ Configuration](/docs/core/TypedBody#configuration).
- **Wrong format.** `tags.Format<"uuid">` rejects non-UUID strings, including UUIDs of the wrong version. `tags.Format<"date-time">` requires full ISO 8601, not just `"2024-01-01"`.
- **Number vs string.** A query parameter typed as `number` will reject `"42"` тАФ but Nestia coerces it automatically on `@TypedQuery`. If it does not, your handler is using `@Query()` not `@TypedQuery()`.

---

### 500 Internal Server Error from `@TypedRoute.*`

The response value didn't match the declared return type. The server logs print the same `{ path, expected, value }` shape тАФ usually a missing field, or `null` where the type doesn't allow it. Fix the return value, or widen the type.

If you'd rather log without throwing (useful during migration), set `stringify: "validate.log"` in the transform options тАФ see [TypedRoute тЖТ Configuration](/docs/core/TypedRoute#configuration).

---

### NestJS DI failing after migrating to `ttsc`

Your `tsconfig.json` likely lost `experimentalDecorators` or `emitDecoratorMetadata`. NestJS needs both for constructor injection тАФ Nestia does not, but NestJS does. Add them back:

```jsonc
{
  "compilerOptions": {
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

---

### Webpack / Rspack: `Cannot find module '@fastify/static'` (or similar optional peer)

NestJS's optional peer dependencies need to be excluded from the bundle. See the `IgnorePlugin` snippet in [Setup тЖТ Legacy тЖТ Single JS file only](/docs/setup/legacy#single-js-file-only).

---

## SDK generation issues

### `npx nestia sdk` outputs an empty directory

Almost always: your `nestia.config.ts` `input` function doesn't include the controllers. Common causes:

- The `AppModule` you imported doesn't transitively import the feature module that hosts the controllers.
- You're using `input: { include: ["..."] }` (glob mode) and the glob misses the files.

Test by importing `AppModule` in a script and listing `app.get(YourController, { strict: false })` тАФ if NestJS can't find it, neither can Nestia.

### Generated SDK uses `unknown` for everything

The transformer didn't see the controller types. Causes:

- The build invoked `tsc`, not `ttsc`.
- The controllers are inside a separate `tsconfig` that isn't what `nestia` reads. Pass `--project path/to/tsconfig.json` to `nestia sdk`.

### Generated SDK imports break in the frontend

The SDK references DTO types by their server-side import path. Two ways to fix:

- Set `clone: true` in `nestia.config.ts` тАФ Nestia clones DTOs into the SDK's own `structures/` directory.
- Move DTOs to a shared package both the server and frontend depend on.

---

## Performance issues

If a single handler is slow, validation is rarely the cause тАФ Nestia's validators run in microseconds. Profile the handler body (DB, network, CPU work) first. Then come back here.

---

## Asking for help

Open an issue at [github.com/samchon/nestia/issues](https://github.com/samchon/nestia/issues) with:

- Output of `npx ttsc --noEmit` (or `npx tsc --noEmit` on legacy).
- The relevant snippet of `tsconfig.json` and `package.json`.
- Your platform (OS, Node version, Go version on `@next`).

That's usually enough to reproduce.
