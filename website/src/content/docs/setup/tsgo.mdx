## Summary

`@nestia/core@next` and `@nestia/sdk@next` use the [`ttsc`](https://github.com/samchon/ttsc) toolchain.

This setup targets TypeScript-Go, the Go migration of the TypeScript compiler. `ttsc` provides the Go-based plugin toolchain, and Nestia ships its transform as a Go binary that `ttsc` loads at compile time. Both `@nestia/core` (controller decorators, request / response validation) and `@nestia/sdk` (SDK / Swagger / e2e generators) are driven by that single binary, with the `typia` transform composed in automatically.

- `ttsc` — build, type-check, watch
- `ttsx` — execute a TypeScript file with type checking
- `@ttsc/unplugin` — run `ttsc` plugins inside bundlers
- `nestia` CLI — generate the SDK, Swagger document, and e2e suite

> [!NOTE]
>
> `@typescript/native-preview` is not a stable TypeScript release yet, and the `@next` line of Nestia is an experimental release that tracks `typia@next`.
>
> Stock `tsc`, `ts-node`, and `tsx` cannot apply the Nestia transform — you must use `ttsc` and `ttsx`.
>
> Go 1.26 or newer must be available on `PATH`, or pointed at by `TTSC_GO_BINARY`. The Nestia transform binary is built on demand from the Go sources shipped inside `@nestia/core`; prebuilt platform binaries are not published in this preview line.
>
> For reproducible builds, pin to an exact dev version (e.g. `typia@13.0.0-dev.20260514`) instead of `@next` — the `@next` dist-tag moves with every dev release.

## Setup

### Install (wizard)

`npx nestia setup` installs every dependency in the right slot (runtime vs dev) and the right order, and then stops. It deliberately does not write `compilerOptions.plugins`, edit `package.json` scripts, modify `nest-cli.json`, or touch `tsconfig.build.json` — `ttsc` auto-discovers the Nestia transform on its own (see [tsconfig.json](#tsconfigjson) below).

After the wizard finishes, replace `nest build` / `nest start` in your scripts with `ttsc` / `node dist/main.js` (see [Compile](#compile) below). If your project uses `tsconfig.build.json`, see the note at the end of the [tsconfig.json](#tsconfigjson) section.

### tsconfig.json

Keep the project strict, and keep the NestJS decorator-metadata flags — NestJS still uses `Reflect.getMetadata("design:paramtypes", …)` for constructor injection. The Nestia transform itself does not require either flag.

```json filename="tsconfig.json" showLineNumbers copy
{
  "compilerOptions": {
    "strict": true,
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true
  }
}
```

That is the entire required configuration. `ttsc` auto-discovers `@nestia/core`, `@nestia/sdk`, and `typia` through their `package.json#ttsc.plugin` markers — you do not need a `compilerOptions.plugins` array unless you want to override transform options (see [Transform Options](#transform-options) below).

> [!NOTE]
>
> `nest new` ships a `tsconfig.build.json` that extends `tsconfig.json` and excludes `node_modules`, `test`, `dist`, and `**/*spec.ts`. Point `ttsc` at it (`ttsc -p tsconfig.build.json` in the build script) so spec files stay out of the production output. Otherwise `ttsc` will compile your `*.spec.ts` alongside your sources.

Wire `ttsc` into the `package.json` scripts the way NestJS projects normally wire `tsc`. For a single watcher that re-runs a script in process, use `ttsx --watch`. For the `nest start --watch` workflow (incremental rebuild + process restart), pair `ttsc --watch` with `node --watch dist/main.js` via `concurrently` so the script behaves the same on cmd, PowerShell, and POSIX shells.

```json filename="package.json" showLineNumbers copy
{
  "scripts": {
    "build": "ttsc",
    "start": "node dist/main.js",
    "start:dev": "concurrently \"ttsc --watch\" \"node --watch dist/main.js\"",
    "start:debug": "concurrently \"ttsc --watch\" \"node --watch --inspect-brk dist/main.js\"",
    "start:script": "ttsx src/main.ts"
  }
}
```

Add `concurrently` as a devDependency (`npm i -D concurrently`). `nest start` and `nest build` from `@nestjs/cli` invoke the stock `tsc` and therefore skip the Nestia transform; replace them with `ttsc` / `ttsx` as shown.

> [!WARNING]
>
> If your `nest-cli.json` sets `compilerOptions.builder` to `"swc"` or `"webpack"`, the Nestia transform never runs — `@nestjs/cli` invokes the chosen builder directly and skips ttsc. Either delete the `builder` field or stop using `nest build` / `nest start`. Replace them with `ttsc` (or `@ttsc/unplugin/webpack` for a Webpack pipeline; see [Bundlers](#bundlers)). In monorepo mode the `builder` field can also live under `projects.<name>.compilerOptions` — check both.

### Monorepo

NestJS monorepos created with `nest g app <name>` put each app at `apps/<name>/src/main.ts` with its own `apps/<name>/tsconfig.app.json`. Build each app with `ttsc -p apps/<name>/tsconfig.app.json`, and run the SDK / Swagger / e2e generators per app — `nestia.config.ts` is project-scoped.

### Nx 23

Fresh Nx 23 Nest workspaces use `@nx/webpack` and an `NxAppWebpackPlugin` build target. `nx serve <app>` runs that Webpack target before starting Node, so `compilerOptions.plugins` alone is not enough; the build must either call `ttsc` directly or load `@ttsc/unplugin/webpack` in the generated Webpack config.

If you keep Nx's Webpack build, install `@ttsc/unplugin` and add the plugin next to `NxAppWebpackPlugin`. In the default Nx 23 layout the command runs from the app directory, so `project: "./tsconfig.app.json"` points at the app config:

```js filename="apps/api/webpack.config.js" showLineNumbers copy
const { NxAppWebpackPlugin } = require("@nx/webpack/app-plugin");
const ttsc = require("@ttsc/unplugin/webpack").default;
const { join } = require("path");

module.exports = {
  output: {
    path: join(__dirname, "../../dist/apps/api"),
    clean: true,
  },
  plugins: [
    new NxAppWebpackPlugin({
      target: "node",
      compiler: "tsc",
      main: "./src/main.ts",
      tsConfig: "./tsconfig.app.json",
      assets: ["./src/assets"],
      optimization: false,
      outputHashing: "none",
      generatePackageJson: true,
      sourceMap: true,
    }),
    ttsc({ project: "./tsconfig.app.json" }),
  ],
};
```

Without that plugin, `nx serve` can build successfully but the first `@TypedRoute` / `@TypedBody` call fails at runtime with "no transform has been configured" because Nx invoked Webpack's stock TypeScript path instead of `ttsc`.

## Generate

First time on this project? Run `npx nestia init` once — it writes a starter `nestia.config.ts` next to your `tsconfig.json`. After that, `npx nestia all` emits the SDK, Swagger document, and e2e suite in one pass.

The `nestia` CLI reads `nestia.config.ts` and the project `tsconfig.json` to emit those artifacts. The Go binary attached to `ttsc` provides the metadata the CLI consumes.

The `nestia` CLI is a thin dispatcher:

- `start`, `template`, `setup` — bootstrap a project from the CLI itself; these run before `@nestia/sdk` is installed.
- `dependencies` — checks installed packages against expected versions; optional.
- `init` — writes a starter `nestia.config.ts`. Run once, before the generators below.
- `sdk`, `swagger`, `e2e`, `all` — drive the project-local `@nestia/sdk` to emit the artifact in their name. `all` runs `sdk + swagger + e2e`.

A fresh `nest new` ships an `AppController` with `@Get`, which Nestia ignores. Replace one method with `@TypedRoute.Get` (see [TypedRoute](/docs/core/TypedRoute)) and re-run `npx nestia all` to see the SDK, Swagger document, and e2e suite populated.

### Coming from `@nestjs/swagger`?

`nestia swagger` generates the OpenAPI document from your TypeScript types — you do not need `@nestjs/swagger` decorators (`@ApiProperty`, `@ApiResponse`, etc.). If you already have them, leave them in place; Nestia ignores them. Migrating from `@nestjs/swagger`? Lift `description`, `title`, `default`, and validation strings from `@ApiProperty` into JSDoc above the same property — Nestia reads JSDoc, so descriptions survive the move. The runtime `SwaggerModule.setup()` call from `@nestjs/swagger` can still be used to serve the generated `swagger.json`, or use [`NestiaSwaggerComposer`](/docs/swagger#runtime-composition) for runtime composition.

## Bundlers

A normal NestJS server compiles with `ttsc` directly — you do not need a bundler. The bundlers below cover two specific cases:

- **Bundling the NestJS server** into a single JS file for serverless or container deployment — Webpack, Rspack.
- **Bundling a frontend** that imports the generated SDK or calls `typia` — Vite, Next.js, Rollup, Rolldown, esbuild, Farm, Bun.build.

> [!WARNING]
>
> **Which row are you?** If you are bundling a NestJS server, only the first two rows work — Webpack with `ts-loader`, or Rspack with `builtin:swc-loader` and decorator-metadata flags. If you are bundling a frontend or library that imports the generated SDK and calls `typia.assert()`, every row works; decorator metadata isn't needed outside the NestJS DI container.
>
> | Bundler | Emits decorator metadata? |
> | --- | --- |
> | **Webpack** with `ts-loader` | Yes — TypeScript compiler honors `emitDecoratorMetadata`. |
> | **Rspack** with `builtin:swc-loader` + decorator flags | Yes, only if you opt in (snippet below). |
> | **Webpack** with community `swc-loader` (no decorator flags) | No — swc strips metadata unless you opt in. |
> | **Rspack** with `builtin:swc-loader` (no decorator flags) | No — same; the snippet below shows the opt-in. |
> | **Vite, Rollup, Rolldown, esbuild, Farm, Bun.build, Next.js** | No — their TS pipeline (esbuild for Vite, SWC for Next.js, Bun's native transform for Bun.build) does not emit decorator metadata. |

### Project

Pass `project` when the bundler should read a different TypeScript config than the one in the project root — for example, when the build `tsconfig.json` targets `commonjs` for Node but the bundle config targets `esnext` for tree-shaking. The path is resolved from the current working directory, and the same key works on every adapter. Three representative shapes follow.

Auto-discovery looks at the `package.json` nearest the selected `tsconfig.json`. Keep `tsconfig.bundle.json` next to `package.json` (the project root), or auto-discovery may not find `@nestia/core` and `typia` in the dependency list. If you must place the bundle config in a subdirectory without its own `package.json`, the parent's `package.json` is still used; if the subdirectory has its own thin `package.json` that does not list nestia, register the plugin explicitly via [Transform Options](#transform-options). Once an entry for that transform appears in `compilerOptions.plugins`, ttsc skips the auto-discovery walk for it, so the bundle-config location no longer matters.

## Transform Options

Register `@nestia/core/native/transform.cjs` explicitly when you want to set transform options (`stringify`, `llm`, …). The other entries in the example below are tombstones / no-ops kept for tooling that scans `compilerOptions.plugins`; you do not need them unless that tooling does. Add `compilerOptions.plugins` only at this point — never as a "you must wire the plugin" step, because auto-discovery already handles that.

```jsonc filename="tsconfig.json" showLineNumbers copy
{
  "compilerOptions": {
    "strict": true,
    "experimentalDecorators": true,
    "emitDecoratorMetadata": true,
    "plugins": [
      { "transform": "typia/lib/transform", "enabled": false },
      { "transform": "@nestia/sdk/lib/transform" },
      {
        "transform": "@nestia/core/native/transform.cjs",
        "stringify": "assert",
        "llm": { "strict": true }
      }
    ]
  }
}
```

The `typia/lib/transform` entry is a tombstone. `ttsc` filters disabled entries before plugin composition, so no extra typia work happens; the entry is there only so package installers that scan `compilerOptions.plugins` still see typia in the chain.

The `@nestia/sdk/lib/transform` entry is a no-op when present without overrides. The Go binary inside `@nestia/core/native/transform.cjs` runs the SDK and typia transforms in a single compiler pass via ttsc's `composes` redirect, so plugin ordering inside the array does not matter.

- `stringify` — pick the serializer used by `@TypedRoute`. Defaults to `"assert"` (validate then stringify). Other values: `"stringify"`, `"is"`, `"validate"`, `"validate.log"`, or `null` to fall back to `JSON.stringify`. See [`TypedRoute`](/docs/core/TypedRoute#configuration).
- `llm` — when set, the transform refuses route or body types that fall outside the schema subset accepted by LLM function calling. `true` forbids tuples; `{ "strict": true }` additionally forbids `additionalProperties` and optional properties.

## Boilerplate

Use `start` for the smaller, SDK-focused starter and `template` for the larger Prisma + PostgreSQL backend with FP / TDD scaffolding.

```bash filename="Terminal" showLineNumbers copy
npx nestia start my-app
npx nestia template my-backend
```

`start` clones, installs, builds, and runs the test suite end-to-end. `template` clones, installs, and builds — leaving the test suite for the first run against your local database / environment, so the wizard does not fail on missing secrets.

If the cloned project is still on the v11 / `ts-patch` toolchain (the templates are shared with the v11 line), follow [Migrating to TypeScript-Go](/docs/setup/legacy#migrating-to-typescript-go) to move it to `ttsc`.

## Troubleshooting

If a build fails with `Error on nestia.core.<method>(): no transform has been configured`, or with a Go diagnostic, run `npx ttsc --noEmit` directly to see the raw stderr. The most common causes are:

- The build was invoked with stock `tsc` (often through a leftover `nest build` script) instead of `ttsc`. Auto-discovery only fires inside `ttsc`.
- `nest-cli.json` has `compilerOptions.builder: "swc"` (or `"webpack"`). `@nestjs/cli` invokes the chosen builder directly, never `ttsc`, and the Nestia transform is skipped. Remove the field or stop using `nest build`.
- `ttsc` is not installed, or `npx nestia setup` was never run.
- Go 1.26 or newer is not on `PATH`, and `TTSC_GO_BINARY` is not pointing at a usable binary.
- You upgraded from v11 but the lockfile still pins old packages — confirm `@nestia/core`, `@nestia/sdk`, `@nestia/fetcher`, `nestia`, and `typia` were all bumped together. Mixed `@latest` / `@next` packages will not interoperate. See [Migrating from Legacy](/docs/setup/legacy#migrating-to-typescript-go).
- You explicitly configured `compilerOptions.plugins` and added another plugin entry that interferes with the Nestia transform's `composes` redirect — drop the explicit array and let auto-discovery handle it.
